# Идеи для дальнейшего ускорения генерации

## Контекст
Базовый фикс уже сделан и запушен в ветку `claude` (коммит `b5ecaaa`):
pinned threadpool на быстрый CPU-кластер устройства + увеличенный
prefill-батч в Director-aware пути. Результат на реальном ходе:
~126с → ~72.5с за ход (в 1.7 раза быстрее), без потери качества.

Ниже — дальнейшие идеи, не реализованы, отсортированы по выгоде/риску.
Каждая требует заметного времени, поэтому отложены на потом.

## 1. KV-cache между ходами (самый большой потенциальный выигрыш)
Сейчас `llama_memory_clear()` в начале каждого вызова стирает весь
контекст, и промпт (~1050 токенов: правила Режиссёра, инструкции, state)
пережёвывается заново на каждом ходе. Статичная часть (правила/грамматика/
примеры, ~300-500 токенов) от хода к ходу не меняется — можно держать её
в KV-кэше постоянно и на каждом ходе гонять заново только динамический
хвост (текущее состояние + действие игрока).

Потенциально экономит добрую половину префилла (сейчас 14.4с) на каждом
ходе, кроме первого. Требует реальной инженерии: логика инвалидации кэша
при реальном изменении состояния, работа с `llama_state_seq_*`/session API
llama.cpp. Самая ценная, но и самая трудоёмкая идея.

## 2. Меньше впустую потраченных попыток Режиссёра
На тестовом ходе ("я осматриваю колонну") модель сделала 5 полных попыток
tool-call (48.7с из 72.5с общего времени) — та же проблема "модель не умеет
вовремя остановиться", которую обсуждали как проблему качества, а не только
скорости. Задача из `task_gbnf_selfcheck.md` (GBNF-грамматика + self-check
pass) должна одновременно почистить калибровку и сэкономить время — каждая
избежанная попытка это ~10с.

## 3. Батчить принудительные (grammar-forced) токены — СДЕЛАНО (частично)
Реализовано 2026-07-14 в `native_bridge.cpp`:
- Общий префикс `<|tool_call>call:director_action{type:"` (идентичен во
  всех 14 ветках грамматики — проверено по `action_rhs()`) теперь
  токенизируется и декодируется одним батчем в `generate_director_action()`
  ДО создания grammar-сэмплера; сам `build_director_action_grammar()`
  больше не включает этот префикс в текст грамматики (root начинается
  сразу с литерала типа).
- `DONE`-ответ вообще не имеет реального выбора модели (name/value/details
  всегда пустые) — теперь собирается одним batch-декодом без сэмплера
  вообще, `create_director_done_sampler()`/`build_director_done_grammar()`
  оставлены в файле неиспользуемыми (мёртвый код, не вызывается).

Проверено на устройстве: грамматика в логах подтверждает отсутствие
префикса (`done ::= "DONE" <[52]> ...`), RAW tool-call собирается и
парсится корректно (`RESULT status=APPLIED`/`NO_CHANGE`), крашей нет.

Не сделано (оставлено на будущее, выше риск/ниже отдача): разделители
`",name:"`/`",value:"`/`",details:"` внутри каждой ветки тоже всегда
литеральны, но батчить их сложнее — нужно детектировать момент закрытия
свободно генерируемого поля (name/value/details) в реальном времени,
без готового API grammar sampler'а для "заглянуть вперёд". Не делали
из-за риска рассинхронизации grammar-состояния при этой сессии.

## 4. ARM-оптимизированный формат кванта — ГОТОВО, действий не требуется
Проверено 2026-07-14: собрали `llama-quantize` из локального чекаута
llama.cpp (`/home/enhort/AndroidStudioProjects/llama.cpp`, коммит
`fdb1db877`) и обнаружили, что явные офлайн-типы `Q4_0_4_8`/`Q4_0_8_8`
из этой версии убраны — `llama-quantize --help` их не перечисляет.

Вместо офлайн-квантования появился runtime repack прямо в
`ggml/src/ggml-cpu/repack.cpp` (строки ~4573-4586): при загрузке модели,
если `ggml_cpu_has_neon() && ggml_cpu_has_matmul_int8()` (т.е. есть i8mm),
обычный `Q4_0` автоматически перекладывается в `q4_0_4x8_q8_0` — это и
есть бывший "Q4_0_4_8", просто теперь поведение, а не отдельный файл.

`/proc/cpuinfo` телефона подтверждает `i8mm` в списке `Features`, а
`CMakeLists.txt` уже собирает .so под `armv8.6-a+dotprod+i8mm`
(сделано в прошлой сессии, комментарий на месте) — значит репак уже
происходит на каждом запуске приложения, бесплатно. Требуется ноль
действий; сам GGUF-файл перегонять не нужно и незачем.
