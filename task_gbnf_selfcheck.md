# Задача: детерминированные guardrails для Director (GBNF + self-check pass)

## Контекст
Проект Pocket D&D (MyDND) — офлайн Android RPG-мастер на локальной LLM
(gemma-4-E2B_q4_0-it.gguf, llama.cpp через JNI). Director (Java/Room) —
источник истины, валидирует и применяет изменения, предложенные моделью.

Известные живые баги, которые НЕ ловятся regex/keyword-логикой в Java:
1. Игрок пишет "я осматриваюсь" → модель вызывает CHECK, хотя в промпте прямо
   указано "осмотр → DONE".
2. Игрок пишет "я беру камень и кладу в карман" (без явного `*item*` маркера)
   → модель корректно предлагает INV_ADD, но ДОПОЛНИТЕЛЬНО пририсовывает
   HP -1 без причины, смену локации и запись в память NPC — то, чего игрок
   не просил.

Это проблема "модель не следует собственным инструкциям", а не нехватка
знаний. Промпт-инструкции и few-shot примеры-якоря снижают частоту, но не
устраняют проблему полностью. Нужны два структурных guardrail поверх текущей
Java-валидации.

Есть диагностический стенд: `BehaviorTestDataset` (11 синтетических сценариев,
лог с тегом `MyDND_BEHAVIOR_TEST`), гоняется напрямую на модели без записи в
Room. Использовать как основной способ проверки "было плохо → стало хорошо".

## Шаг 0 (обязательно перед кодом)
Прежде чем писать грамматику или self-check промпт, изучи в текущем коде:
- Точный JSON/tool-call формат, который сейчас генерирует директор
  (`DirectorToolSpec` и то, что реально уходит в модель и возвращается из неё).
- Полный список action-типов и их полей (тех самых ~15 типов: INV_ADD,
  HEALTH_DAMAGE, LOCATION, MONEY, QUEST_*, NPC_*, WORLD_*, ABILITY_*, EFFECT_*
  и т.д.) — с точными именами enum/строк как в коде, не как в этом описании.
- Как сейчас вызывается llama.cpp через JNI (есть ли уже путь для передачи
  grammar/sampling параметров в generate-вызов, или его нужно добавить).

Прежде чем менять код, предложи на ревью:
- Черновик GBNF-грамматики (или альтернативный constrained-decoding подход,
  если GBNF не ложится на текущий формат вывода).
- Черновик rule-check промпта для задачи 2 (список правил + формат ответа).

## Задача 1 — GBNF-грамматика для структуры вывода директора

Цель: сделать физически невозможным для модели сгенерировать вывод, который
нарушает структурные правила, вместо того чтобы полагаться на то, что модель
"поймёт" инструкцию в промпте.

Обязательные ограничения на уровне грамматики:
- Поле intent/decision (как бы оно ни называлось в реальной схеме) — строгий
  enum, не свободный текст.
- Если intent = DONE (или эквивалент "ничего не менять") — список действий
  ДОЛЖЕН быть пустым массивом. Грамматика не должна допускать непустой список
  действий при этом intent.
- Если intent = CHECK — список действий ограничен тем, что уместно для
  проверки (без произвольных HP/LOCATION/NPC правок, если это не relevant).
- Структурный cap на количество действий одного типа за ход — перенести уже
  существующий Java-лимит (1 HP-действие, 1 MONEY-действие за ход из
  `TurnPolicy`) на уровень грамматики, если формат вывода это позволяет —
  цель дать модели меньше шансов сгенерировать лишнее, а не только отфильтровать
  постфактум в Java.
- Валидный JSON/tool-call формат гарантирован грамматикой (не полагаться на
  то, что модель не сломает синтаксис).

Технически:
- Разместить .gbnf файл(ы) в assets проекта.
- Подключить через существующий JNI-вызов llama.cpp (использовать
  llama_grammar API/эквивалент — уточнить как это делается в текущей версии
  llama.cpp, зашитой в проект).
- Продумать fallback: что происходит, если constrained generation даёт
  ошибку/таймаут/деградацию качества — не должно ронять партию.

Приёмка:
- Тестовый сценарий "я осматриваюсь" из `BehaviorTestDataset` — до фикса
  падает (даёт CHECK), после — стабильно даёт DONE с пустым списком действий.
- Существующие 11 unit-тестов (`InventoryActionHintResolverTest`,
  `DirectorFlowControllerTest`) остаются зелёными.
- Новые unit/instrumented-тесты, покрывающие оба живых бага из раздела
  "Контекст".

## Задача 2 — Self-check проход перед применением изменений

Цель: второй, дешёвый LLM-вызов, который проверяет уже предложенный список
действий на соответствие явным правилам ДО того, как Java применит их к Room —
ловит нарушения, которые грамматика структурно не покрывает (например,
"HP-изменение без явной причины в тексте игрока").

Требования:
- Формализовать в виде короткого списка проверяемых правил (взять из уже
  существующих: `TurnPolicy`, инструкции в `DirectorToolSpec.playerActionRules`,
  плюс два новых правила против конкретных живых багов).
- Второй вызов модели: на вход — предложенный список действий + исходное
  сообщение игрока + список правил; на выход — по каждому подозрительному
  действию ALLOW/REJECT с кратким основанием (не свободный текст-эссе,
  минимальный формат, короткий maxTokens для скорости).
- При REJECT — вырезается только конкретное нарушающее действие, а не весь ход
  целиком (сохранить принцип точечной хирургии, уже принятый в проекте для
  Undo).
- Должно быть переключаемо (feature flag / настройка), т.к. добавляет
  latency на устройстве — нужно дать возможность сравнить производительность
  с/без на реальном устройстве.
- Логирование решений self-check с отдельным тегом (по аналогии с
  `MyDND_BEHAVIOR_TEST`), чтобы можно было анализировать, что и почему
  отклоняется.

Приёмка:
- Тестовый сценарий "беру камень и кладу в карман" — до фикса модель добавляет
  лишний HP/LOCATION/NPC, после — self-check отклоняет лишние действия,
  INV_ADD проходит.
- Новые unit-тесты на self-check логику (мокать ответ второй LLM, проверять
  что Java корректно вырезает только отклонённые действия).
- Замер latency-оверхеда на реальном устройстве (примерная цифра в PR-описании).

## Общие ограничения
- Language-agnostic — никакой привязки к конкретному естественному языку
  (текущий принцип проекта, уже закреплён для инвентаря/HP/money-логики).
- Не ломать существующие 11 зелёных unit-тестов и текущий Undo-механизм
  (HP/LOCATION/INV/NPC memory).
- Ветка `claude`, стиль коммитов как в существующей истории
  (короткий императивный заголовок + при необходимости body).
- Обе задачи можно делать и коммитить независимо друг от друга — задача 1
  первая по приоритету (дешевле, детерминированнее), задача 2 вторая.
