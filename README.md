# Pocket D&D / MyDND

**Pocket D&D** — локальный офлайн RPG-мастер для Android. Приложение написано на Java/XML, хранит игру в Room/SQLite и запускает локальную GGUF-модель через llama.cpp/JNI.

## Главный принцип

```text
LLM
├── повествование
├── атмосфера
├── диалоги NPC
└── понимание естественного языка

Java + Room
├── инвентарь
├── состояние персонажа
├── NPC
├── кубики
├── журнал
├── события мира
└── долговременная память
```

LLM не является базой данных. Каноническое состояние хранит приложение.

## Текущий рабочий стек

- Android;
- Java/XML;
- Room / SQLite;
- llama.cpp Android JNI/CMake/NDK;
- локальная модель `gemma-4-E2B_q4_0-it.gguf`;
- офлайн-работа на телефоне.

## Что уже работает

- создание миров, живых историй мира, персонажей и кампаний;
- локальная генерация мастера со streaming и отменой;
- one-pass inventory tool cycle в одном llama_context;
- Java-валидация ADD / REMOVE / NO CHANGE;
- журнал и сохранение кампаний;
- живой мир и `WORLD EVENT` раз в 5 завершённых ответов мастера;
- `WORLD SUMMARY` примерно раз в 30 мировых ходов или после события важности 3;
- случайные фоновые события мира;
- базовые характеристики героя: STR / DEX / INT / CHA / HP;
- базовое состояние NPC: HP, скрытое отношение, локация, знания;
- DiceService: d4, d6, d8, d10, d12, d20, d100;
- кнопка полного сброса игровых данных без удаления модели и настроек;
- универсальные плашки изменений состояния.

## Плашки изменений состояния

Вероятностная модель может ошибаться, поэтому важные изменения не должны быть скрыты.

Сейчас приложение показывает отдельные плашки для:

- получения предмета;
- потери предмета;
- важного воспоминания NPC;
- необходимости проверки кубика;
- фонового изменения живого мира.

Безопасные изменения можно отменить. Отменённая плашка становится серой, а состояние Room откатывается.

Не все плашки имеют Undo: действие доступно только там, где откат безопасен и однозначен.

## Инвентарь

Для более точной работы с предметами можно выделять название звёздочками:

```text
Я беру *железный ключ*.
Я выбрасываю *старый меч*.
```

Это подсказка, а не обязательный синтаксис.

## NPC memory

Мастер может отметить, что конкретный активный NPC надолго запомнил поступок игрока.

Игрок видит:

```text
КУЗНЕЦ ЗАПОМНИЛ
Ты помог его дочери
```

Числовое отношение NPC (`-10 ... +10`) остаётся скрытым и хранится только в базе.

## Проверки

Если рискованное действие требует проверки, мастер может создать плашку:

```text
НУЖНА ПРОВЕРКА
ЛОВКОСТЬ • СЛ 12
[ БРОСИТЬ d20 ]
```

Результат броска сохраняется и попадает в контекст следующего хода.

## Живой мир

Мир имеет общий `world_turn_count`, принадлежащий `WorldTimeline`, а не отдельному герою.

```text
5 ответов MASTER
→ WORLD EVENT extraction

около 30 мировых ходов
→ WORLD SUMMARY

редкий случайный интервал
→ RANDOM WORLD EVENT
```

Новый герой в той же истории мира получает уже изменившийся мир.

## Память

Не надо держать всю кампанию в контексте модели.

В prompt передаются только:

- текущая сцена;
- состояние мира;
- состояние героя;
- активные NPC;
- активные ситуации;
- последние события;
- релевантные факты;
- текущий инвентарь.

## Правила разработки

- Java — основной язык;
- не делать один гигантский модуль правил;
- LLM предлагает, Java валидирует;
- Room — источник истины;
- сначала минимально рабочая механика, затем усложнение;
- проект должен работать полностью локально на телефоне.

## Ближайшие этапы

1. Дальнейшая проверка плашек и Undo.
2. Квесты через общую Event/StateChange-систему.
3. Локации, принадлежащие World / WorldTimeline.
4. Полноценные изменения HP и NPC-state через те же плашки.
5. Экспорт кампании в JSON.
6. Полировка UI.
